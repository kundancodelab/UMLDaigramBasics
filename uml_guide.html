UML Diagrams for iOS Developers

A simple, practical guide on how to design and read diagrams for Swift mobile apps.

Think of UML as a visual representation of your code. If you can write Swift, you can understand UML.

We will focus on only what you need to build great iOS apps, starting with the two most important diagrams:

  • The Class Diagram (how your code is structured)
  • The Sequence Diagram (how your code runs over time)

Part 1: The Class Diagram (The Structure)

A Class Diagram is a blueprint of the classes, structs, and protocols in your app, showing how they connect.

1. The Class Box

In UML, every Class or Struct is represented by a box divided into three sections:

┌────────────────────────────────────────┐
│               ClassName                │  <-- 1. Name of Class
├────────────────────────────────────────┤
│ - privateProperty: String              │  <-- 2. Variables / State
│ + publicProperty: Int                  │
├────────────────────────────────────────┤
│ + fetchData()                          │  <-- 3. Functions / Actions
│ - validateInput() -> Bool              │
└────────────────────────────────────────┘

The Secret Code (Access Modifiers):

Before any variable or function, you place a symbol to show its visibility:

  • - (Minus) = Private. Only this class can see/use it. (e.g., - modelContext)
  • + (Plus) = Public / Internal. Other classes can access it. (e.g., + startSync())

2. The Relationships (How boxes connect)

To connect these boxes, we use lines and arrows. Here are the two relationships you need to know:

Relationship A: Association ("Has a")

One class holds a reference to another class.

  • Swift Code: let repository: BugRepository
  • UML: A solid line with an arrow pointing to the class it holds.
classDiagram
class BugReportCreationViewModel {
    -repository: BugReportingRepository
    +advance()
}
class BugReportingRepository {
    +createTicket()
}
BugReportCreationViewModel --> BugReportingRepository : holds reference to

Relationship B: Conformance ("Is a" Protocol)

A class conforms to a Protocol.

  • Swift Code: struct BugReportingRepository: BugReportingRepositoryProtocol
  • UML: A dashed line with an empty triangle head pointing to the Protocol.
classDiagram
class BugReportingRepositoryProtocol {
    +createTicket()
}
class BugReportingRepository {
    +createTicket()
}
BugReportingRepository ..|> BugReportingRepositoryProtocol : conforms to

Part 2: The Sequence Diagram (The Flow)

If the Class Diagram is a map of a city, the Sequence Diagram is a video showing a car driving through the streets. It shows how objects talk to each other over time.

How to Read a Sequence Diagram:

  1. Objects are represented by boxes at the top.
  2. Dotted vertical lines go down from each object (called Lifelines).
  3. Time flows from top to bottom.
  4. Horizontal arrows show messages (function calls) sent between objects.

A Simple iOS Example: User Taps "Submit Bug"

Here is how code flows when a user taps submit:

sequenceDiagram
    actor User
    participant View as BugView (SwiftUI)
    participant VM as BugViewModel
    participant Repo as BugRepository

    User->>View: Taps "Submit" Button
    View->>VM: VM.submitTicket()
    Note over VM: Validates input...
    VM->>Repo: Repo.createTicket(request)
    Repo-->>VM: Returns Ticket ID ("BUG-123")
    VM-->>View: Show Success Alert
    View-->>User: Visual feedback on screen
  • Solid arrow (->>): Calling a function.
  • Dashed arrow (-->>): Returning a value (or callback/completion handler).

The Golden Mindset Shift: Draw first, then code. Drawing a diagram before coding forces you to catch design mistakes when they are cheap to fix — on paper — rather than after you have written 300 lines of code.
